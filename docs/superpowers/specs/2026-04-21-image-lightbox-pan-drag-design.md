# Image Lightbox Pan-by-Drag Design

**Date:** 2026-04-21
**Status:** Proposed
**Scope:** Enable the user to pan (drag) an image inside the fullscreen lightbox at any zoom level.

## Problem

The lightbox (`src/tools/image/ui.ts::openLightbox`) supports zoom via toolbar buttons and `+`/`-` keys but offers no way to pan the zoomed image. At zoom > 1 the user cannot reach parts of the image that fall outside the viewport. Any click on the dialog outside the toolbar currently closes the lightbox, leaving no drag affordance.

## Goal

Let the user drag the image inside the lightbox to shift its displayed position. Works at any zoom level, with mouse, touch, and pen. Bounds prevent losing the image off-screen. Click-to-close still works when the user clicks without dragging.

## Non-Goals

- Pinch-to-zoom gestures (separate feature).
- Inertia / momentum scrolling.
- Keyboard panning.
- Saving pan/zoom state between lightbox sessions.

## User-Visible Behavior

1. Opening the lightbox shows the image centered (unchanged).
2. The dialog cursor is `grab`. During an active drag the cursor is `grabbing`.
3. Pointer-down + move > 3px anywhere on the dialog (except toolbar) starts a drag. Releasing after drag does NOT close the lightbox.
4. Pointer-down + release without exceeding the 3px threshold is treated as a click and closes the lightbox (current behavior).
5. Pan is clamped so the image center stays within the viewport on both axes (the image can never be dragged so its center is off-screen).
6. Calling `setZoom(1)` (reset button, `-` key reaching 1, etc.) snaps pan back to `{0, 0}`.
7. Touch drag on mobile pans the image (does not scroll the page).

## Architecture

### State

```ts
const zoomState = { value: 1 };
const panState  = { x: 0, y: 0 };
```

### Transform composition

Replace:
```ts
img.style.transform = `scale(${zoomState.value})`;
```
with:
```ts
img.style.transform = `translate(${panState.x}px, ${panState.y}px) scale(${zoomState.value})`;
```
Apply from a single `applyTransform()` helper called by `setZoom` and pan updates.

### setZoom integration

`setZoom(next)` clamps to `[ZOOM_MIN, ZOOM_MAX]`. After clamping:
- If clamped value === 1 → reset `panState` to `{0, 0}`.
- Otherwise re-clamp existing pan to the new bounds (so zooming out does not leave pan outside new max).
- Call `applyTransform()` and `syncResetLabel()`.

### Drag handlers

On the `dialog` element:

- `pointerdown`:
  - Ignore if `event.target` is inside `toolbar`.
  - Record `start = { x: event.clientX, y: event.clientY }` and `origin = { ...panState }`.
  - Set `pointerDown = true`, `dragging = false`.
  - Call `dialog.setPointerCapture(event.pointerId)`.

- `pointermove`:
  - Only when `pointerDown`.
  - `dx = event.clientX - start.x`, `dy = event.clientY - start.y`.
  - If not yet dragging and `Math.hypot(dx, dy) > DRAG_THRESHOLD (3)` → mark `dragging = true`, set cursor to `grabbing`.
  - If dragging, update `panState = clamp({ x: origin.x + dx, y: origin.y + dy })`, call `applyTransform()`.

- `pointerup` / `pointercancel`:
  - Reset cursor to `grab`.
  - If `dragging`, swallow the next `click` event on the dialog so click-to-close does not fire.
  - Reset `pointerDown`, `dragging`.

### Click-to-close coexistence

The existing `dialog.click` handler closes the lightbox. After a drag we install a one-shot capture listener:
```ts
const swallow = (e: MouseEvent) => {
  e.stopPropagation();
  dialog.removeEventListener('click', swallow, true);
};
dialog.addEventListener('click', swallow, true);
```
This runs before the close handler and cancels only the click generated by the current drag release. Normal clicks without drag hit the existing close handler unchanged.

### Bounds clamp

```ts
function clampPan(p: { x: number; y: number }): { x: number; y: number } {
  const rect = img.getBoundingClientRect();
  // rect already reflects current scale.
  const maxX = rect.width / 2;
  const maxY = rect.height / 2;
  return {
    x: Math.max(-maxX, Math.min(maxX, p.x)),
    y: Math.max(-maxY, Math.min(maxY, p.y)),
  };
}
```
Uses `getBoundingClientRect()` so the clamp adapts automatically to current scale. Image center stays within viewport because the viewport (dialog) is always at least as wide as the image center offset at zoom 1 (image fits). At higher zoom the half-dimensions grow, giving more room to pan.

### Styles

- `.blok-image-lightbox` → `cursor: grab; touch-action: none; user-select: none;`
- `.blok-image-lightbox.is-dragging` → `cursor: grabbing;`
- `.blok-image-lightbox__image` → `pointer-events: none;` so pointer events always target the dialog (simplifies handler wiring).

Toolbar continues stopping propagation, so interacting with toolbar buttons never starts a drag.

## Testing (TDD)

Unit tests in `test/unit/tools/image/ui.test.ts` (extend or create). All tests use jsdom + manual `PointerEvent` dispatch.

1. **Drag translates image.** openLightbox → dispatch `pointerdown` at (100,100), `pointermove` at (150,140), `pointerup`. Expect `img.style.transform` to contain `translate(50px, 40px)`.
2. **Click without drag closes.** `pointerdown` + `pointerup` at same coords + subsequent `click` → dialog removed from DOM.
3. **Drag then release does not close.** Drag >3px, release → dialog still in DOM after emitted `click`.
4. **Pan clamped at bounds.** Set large delta (e.g., 99999px). Final pan equals `±rect.width/2` / `±rect.height/2`.
5. **Zoom reset snaps pan.** Drag to pan, then call reset button (or setZoom to 1). Expect `transform` to be `translate(0px, 0px) scale(1)`.
6. **Zoom out re-clamps pan.** Pan to bound at scale 2, setZoom(1.25). Pan reduced to new max bound (not preserved raw).
7. **Toolbar click does not start drag.** `pointerdown` on a toolbar button does not initialize drag state; clicking the button triggers its action.
8. **Below-threshold move still closes.** Move 2px then release → treated as click, dialog closes.

## Open Questions

None — all answered during brainstorming (drag always active, clamp to viewport center, pointer events for touch support, snap pan on reset to 1x).

## Risks

- **Pointer capture edge cases**: dragging and releasing outside the viewport must still end the drag. Pointer capture on dialog handles this; tests cover pointercancel.
- **Flaky jsdom geometry**: `getBoundingClientRect()` returns zeros in jsdom unless stubbed. Tests stub `img.getBoundingClientRect` to return a fixed rect.
- **CSS regression**: `pointer-events: none` on image removes hover affordances the lightbox currently relies on — audit before landing. (No current hover behavior on the lightbox image, so safe.)
